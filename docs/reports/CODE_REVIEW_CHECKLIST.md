# Video-to-Action 代码审查检查清单

**使用说明**: 审查者在每次代码审查时使用此清单，确保不遗漏关键检查项。

---

## 审查前准备

- [ ] 已阅读变更说明和相关 Issue/PR 描述
- [ ] 已理解变更的意图和影响范围
- [ ] 已检查 CI/CD 流水线状态（全部通过）

---

## 1. 自动化检查（必须全部通过）

- [ ] **代码格式化** - `black` 和 `isort` 已运行
- [ ] **类型检查** - `mypy` 检查通过，无类型错误
- [ ] **代码质量** - `pylint` 或 `flake8` 评分可接受
- [ ] **测试覆盖** - 所有测试通过，覆盖率未下降
- [ ] **安全扫描** - 无敏感信息泄露（API密钥、密码等）

**工具命令**:
```bash
# 格式化
black video_to_action/ tests/
isort video_to_action/ tests/

# 类型检查
mypy video_to_action/

# 代码质量
pylint video_to_action/

# 测试
pytest tests/ --cov=video_to_action --cov-report=term-missing
```

## 2. 功能正确性

### 核心检查

- [ ] 代码实现了预期功能，符合需求
- [ ] 处理了所有边界条件（空值、空列表、None、最大值/最小值）
- [ ] 错误处理恰当，使用异常而非返回错误字典
- [ ] 异步代码正确使用了 `await` 和 `async`
- [ ] 无逻辑错误（如错误的条件判断、off-by-one错误）

### 项目特定

- [ ] 视频下载支持所有声称的平台（抖音、B站、YouTube）
- [ ] 配置文件键名拼写正确（注意 `safety` 而非 `safety`）
- [ ] LLM 调用有超时和重试机制
- [ ] 数据库操作使用参数化查询

---

## 3. 安全性 🔴

### 必须检查（发现问题即为 Blockers）

- [ ] **无命令注入** - 不使用 `os.system()` 或 `subprocess` 拼接用户输入
- [ ] **无SQL注入** - 使用参数化查询
- [ ] **无路径遍历** - 验证文件路径在允许范围内
- [ ] **无敏感信息** - 未提交 API Key、密码、Token
- [ ] **输入验证** - 所有外部输入都经过验证和清理
- [ ] **权限检查** - 敏感操作有适当的权限验证

### 安全检查代码示例

```python
# ✅ 安全：使用 subprocess 参数列表
subprocess.run(["ffmpeg", "-i", video_path, "-o", output_path])

# ✅ 安全：验证文件路径
output_dir = Path("outputs")
file_path = output_dir / user_filename
if not file_path.resolve().startswith(output_dir.resolve()):
    raise ValueError("非法的文件路径")

# ✅ 安全：使用环境变量
API_KEY = os.getenv("OPENAI_API_KEY")
```

## 4. 可维护性

### 命名与结构

- [ ] 命名清晰、一致，符合 PEP 8
- [ ] 函数长度 < 50 行（超过应拆分）
- [ ] 圈复杂度 < 10（超过应重构）
- [ ] 无重复代码（应抽取为函数/类）
- [ ] 模块职责单一，符合单一职责原则

### 文档

- [ ] 所有公共函数/类有文档字符串（Google风格）
- [ ] 文档字符串包含 Args、Returns、Raises（如适用）
- [ ] 复杂逻辑有注释解释"为什么"（而非"是什么"）
- [ ] README 和架构文档已更新（如适用）

---

## 5. 性能

- [ ] 无明显的 O(n²) 或更高复杂度的算法
- [ ] 无不必要的内存拷贝或大对象复制
- [ ] 数据库查询无 N+1 问题
- [ ] 循环中无不变的计算（应提取到循环外）
- [ ] 合理使用缓存（如适用）
- [ ] 文件和网络资源及时关闭（使用 `with` 语句）

---

## 6. 测试

### 测试覆盖

- [ ] 新代码有对应的单元测试
- [ ] 测试覆盖了关键路径和边界条件
- [ ] 测试名称描述性强（如 `test_process_video_with_invalid_url_raises_error`）
- [ ] 测试相互独立，不依赖执行顺序
- [ ] 合理使用 Mock 和 Fixture

### 测试质量

- [ ] 断言明确，错误信息清晰
- [ ] 无跳过的测试（除非有充分理由）
- [ ] 测试数据合理，覆盖正例和反例
- [ ] 集成测试覆盖了关键的用户场景

## 7. Python 特定规范

### 类型注解

- [ ] 所有函数参数和返回值有类型注解
- [ ] 使用 `typing` 模块的类型（如 `Optional`, `List`, `Dict`）
- [ ] 复杂类型使用 `TypeAlias` 或 `NewType` 提高可读性

### 代码风格

- [ ] 行长度 ≤ 120 字符
- [ ] 导入顺序正确（标准库 → 第三方 → 本地模块）
- [ ] 使用 f-string 而非 `%` 或 `.format()`
- [ ] 使用列表推导式而非显式循环（如适用且可读）
- [ ] 使用上下文管理器（`with` 语句）管理资源

## 8. 项目特定检查

### 配置管理

- [ ] 配置键名拼写正确（检查 `safety` 配置）
- [ ] 敏感配置通过环境变量读取
- [ ] 配置有合理的默认值
- [ ] 配置验证在加载时完成

### 错误处理

- [ ] 使用自定义异常类（`VideoToActionError` 及其子类）
- [ ] 异常链使用 `raise ... from ...`
- [ ] 日志记录了足够的错误上下文
- [ ] 向用户提供了清晰的错误消息

### 日志记录

- [ ] 使用 `logging` 模块而非 `print()`

- [ ] 关键流程和决策点有日志

## 9. 审查评论规范

### 评论格式

使用以下格式撰写审查评论：

```
### 🔴/🟡/💭 **[Category]: [Title]**

**File:** `path/to/file.py`, **Line:** XX

**Problem:** [描述问题]

**Why:** [解释为什么这是问题]

**Suggestion:**
[提供具体的修复建议或代码示例]
```

### 严重程度标记

- 🔴 **Blocker** - 必须修复才能合并
  - 安全漏洞、数据丢失风险、破坏API契约
  
- 🟡 **Suggestion** - 应该修复，但不阻断合并
  - 缺少测试、命名不清晰、性能问题
  
- 💭 **Nit** - 可以后续改进，不紧急
  - 风格细节、文档缺失、次要优化
